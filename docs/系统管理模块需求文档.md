## 一、模块概述

### 1.1 背景与目标

交易所后台管理系统需要一套完整的**基于角色的权限控制（RBAC）体系**，实现管理员身份的分层管理与权限精细化控制。

本模块解决的核心问题：
- 限制超级管理员权限（不暴露给普通运营）
- 实现多级管理员体系，支持权限继承与限制
- 支持权限交接，确保人员变动时业务连续性
- 软删除机制，保留操作审计记录

### 1.2 术语定义

| 术语 | 定义 |
|---|---|
| 超级管理员（Super Admin） | 系统最高权限者，仅在初始化时创建，不可删除 |
| 管理员（Admin） | 由超级管理员创建，拥有部分权限，可创建下级用户 |
| 下级用户（Sub User） | 由管理员创建，拥有管理员授予的子集权限 |
| 权限节点（Permission） | 系统中最小的权限单位，如"用户查看"、"订单删除" |
| 权限组（Role） | 多个权限节点的集合，如"运营主管" |
| 交接（Transfer） | 将失效用户的权限转嫁给另一个有效用户 |

## 二、角色体系设计

### 2.1 角色层级

```
超级管理员（Super Admin）
    └── 管理员（Admin）Level 1
            └── 下级用户（Sub User）Level 2
                    └── （3级封顶，不支持再往下创建）
```

> **层级规则**：最多3级。超级管理员为Level 0，管理员为Level 1，下级用户为Level 2。
> 下级用户不可再创建下级用户（Level 2为叶子节点）。

### 2.2 角色对比

| 特性 | 超级管理员 | 管理员 | 下级用户 |
|---|---|---|---|
| 数量 | 仅1个 | 多个 | 多个 |
| 创建方式 | 后端启动时自动初始化 | 超级管理员创建 | 管理员创建 |
| 可否删除 | ❌ 不可删除 | ✅ 可删除（软删除，不可恢复） | ✅ 可删除（软删除，不可恢复） |
| 可否给自己修改权限 | ❌ 不可修改自己的权限 | ❌ 只能修改下级 | ❌ 只能修改下级 |
| 可修改自己的信息 | 头像、昵称、邮箱、手机号、密码 | 头像、昵称、邮箱、手机号、密码 | 头像、昵称、邮箱、手机号、密码 |
| 可授权范围 | 所有权限 | 仅自己拥有的权限 | 仅自己拥有的权限 |
| 交接审批 | 无需审批（自己是最高权限） | 需超级管理员审批 | 需直属管理员审批 |
| 权限上限 | 无限制 | 超级管理员授予的范围 | 上级管理员授予的范围 |

## 三、功能详情

### 3.1 超级管理员初始化

#### 触发条件
- 后端服务**启动时**自动检测数据库中是否已有超级管理员
- 若无，自动创建初始超级管理员账号
- **启动时仅能创建一次**，后续启动若已存在则跳过

#### 初始化数据
- 账号：`admin`（或自定义）
- 密码：（强密码要求：12位以上，大小写+数字+特殊字符）
- 权限：**全部权限**
- 状态：**启用**

#### 安全要求
- 初始化完成后，首次登录强制要求修改默认密码
- 初始化过程记录启动日志
- 建议：仅内网/服务器本地启动时有网络连接可访问

### 3.2 管理员管理（超级管理员操作）

#### 3.2.1 创建管理员

**输入字段：**

| 字段 | 说明 | 必填 | 校验 |
|---|---|---|---|
| 用户名 | 登录账号，唯一 | ✅ | 4-20位，字母/数字/下划线 |
| 姓名 | 真实姓名 | ✅ | 2-20字 |
| 手机号 | 联系方式 | ✅ | 手机号格式 |
| 邮箱 | 通知邮箱 | ✅ | 邮箱格式 |
| 密码 | 初始密码 | ✅ | 强密码规则 |
| 权限组 | 从预设权限组中选择 | ✅ | 不可超过超级管理员范围 |
| 生效时间 | 定时启用（可选） | ❌ | 需晚于当前时间 |
| 失效时间 | 定时禁用（可选） | ❌ | 需晚于生效时间 |

**权限范围限制：**
- 超级管理员在创建管理员时，可选"全部权限"或指定权限组
- 不允许授予超级管理员不具备的权限

#### 3.2.2 修改管理员

可修改字段：姓名、手机号、邮箱、权限组、生效/失效时间、状态（启用/禁用）

**禁止行为：**
- 管理员不可修改自己的权限（需超级管理员操作）
- 管理员修改下级用户的权限时，不能超出他自己的权限

#### 3.2.3 删除（禁用）管理员

**操作类型：**

| 类型 | 说明 |
|---|---|
| 直接禁用 | 直接禁用该管理员，所有权限立即失效 |
| 交接后禁用 | 将权限交接给其他管理员/下级用户后，原管理员自动禁用 |

**⚠️ 禁用后不可恢复**：禁用后该账号永久失效，不可重新启用。如需恢复，只能重建账号并重新分配权限。

**软删除实现：**
- `status` 字段设为 `disabled`
- `deleted_at` 记录删除时间
- 数据库保留记录，不做物理删除
- 该管理员用户名不可被新管理员复用（用户名全局唯一）

#### 3.2.4 管理员列表

| 字段 | 说明 |
|---|---|
| 用户名 | 登录账号 |
| 昵称 | 在后台显示的名称 |
| 手机号 | 联系方式（脱敏展示：138****1234） |
| 邮箱 | 通知邮箱 |
| 权限组 | 所属权限组名称 |
| 下级用户数 | 该管理员创建的下级用户数量 |
| 状态 | 启用 / 禁用 / 已过期 |
| 创建时间 | 账号创建时间 |
| 最后登录 | 最近一次登录时间 |

支持筛选：状态、权限组、创建时间范围

### 3.5 权限交接（Transfer）

#### 3.5.1 触发场景

- 管理员主动申请离职（触发交接）
- 管理员意外离职（超级管理员主动发起）
- 下级用户主动申请交接（需其直属上级审批）

#### 3.5.2 审批层级

| 交接类型 | 发起人 | 审批人 | 执行条件 |
|---|---|---|---|
| 超级管理员交接（不可操作） | — | — | 超级管理员不可删除，不存在交接 |
| 管理员（Level 1）交接 | 管理员本人或超级管理员 | **超级管理员** | 审批通过后方可执行 |
| 下级用户（Level 2）交接 | 下级用户本人或其上级 | **直属管理员（Level 1）** | 审批通过后方可执行 |

> **示例**：Level 1管理员A离职，权限交接给Level 1管理员B → 需超级管理员审批
> **示例**：Level 2下级用户C离职，权限交接给Level 2下级用户D → 需C的直属上级管理员A审批

#### 3.5.2 交接流程

```
1. 交接发起人选择被交接者（A）和接收者（B）
2. 系统展示：
   - A持有的权限列表
   - B已有的权限列表
   - 将新增到B的权限（A有而B没有的权限）
   - B已拥有但A也有的权限（不变）
3. 发起人提交交接申请 → 进入待审批状态
4. 审批人审核（确认权限范围是否合规）
   - 审批通过：执行交接，A状态变为 disabled，B权限累加
   - 审批拒绝：交接取消，A权限不变
5. 记录交接日志（发起人、审批人、接收人、时间、权限变更明细）
```

#### 3.5.3 权限冲突处理

**场景1：B的权限是A的子集**
→ 简单合并，直接累加

**场景2：B的权限与A有交叉但互有有无**
→ 仅新增A有而B没有的权限，A原有的不重复叠加

**场景3：B的权限包含A的全部**
→ A删除后B权限不变，仅记录交接日志

**场景4：B的权限与A完全不重叠**
→ 允许，仅做简单累加

#### 3.5.4 交接限制

- 超级管理员不可被交接（不可删除，不可禁用）
- 交接记录永久保留，不可篡改
- 交接过程中，被交接者（A）权限暂不变化，直到审批通过
- 交接审批超时（72小时未审批）自动取消

### 3.7 个人中心（全员通用）

所有管理员/下级用户均可修改自己的以下信息：

| 字段 | 说明 | 校验 |

支持筛选：时间范围、操作类型、操作人、目标对象

## 四、数据模型

### 4.1 管理员表（admins）

```sql
CREATE TABLE admins (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    username        VARCHAR(20)  NOT NULL UNIQUE,  -- 登录账号
    password_hash   VARCHAR(255) NOT NULL,         -- bcrypt加密
    real_name       VARCHAR(20)  NOT NULL,         -- 真实姓名
    phone           VARCHAR(20)  NOT NULL,         -- 手机号
    email           VARCHAR(100) NOT NULL,         -- 邮箱
    role_id         BIGINT      NOT NULL,         -- 关联roles表
    parent_id       BIGINT      DEFAULT NULL,     -- 上级管理员ID（NULL=超级管理员）
    level           TINYINT     NOT NULL DEFAULT 1, -- 1=管理员, 2=下级用户
    status          TINYINT     NOT NULL DEFAULT 1, -- 1=启用, 0=禁用
    deleted_at      DATETIME    DEFAULT NULL,      -- 软删除时间
    created_at      DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    last_login_at   DATETIME    DEFAULT NULL,
    last_login_ip   VARCHAR(45) DEFAULT NULL,
    INDEX idx_parent (parent_id),
    INDEX idx_status (status),
    INDEX idx_deleted (deleted_at)
);
```

### 4.2 角色表（roles）

```sql
CREATE TABLE roles (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    name          VARCHAR(20)  NOT NULL UNIQUE,
    description   VARCHAR(200),
    is_preset     TINYINT      NOT NULL DEFAULT 0,  -- 1=系统预设不可删
    created_at    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 4.3 权限节点表（permissions）

```sql
CREATE TABLE permissions (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    code          VARCHAR(50)  NOT NULL UNIQUE,  -- 如: user_view
    name          VARCHAR(30)  NOT NULL,           -- 如: 用户查看
    module        VARCHAR(30)  NOT NULL,           -- 如: 用户管理
    created_at    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### 4.4 角色-权限关联表（role_permissions）

```sql
CREATE TABLE role_permissions (
    role_id         BIGINT NOT NULL,
    permission_id   BIGINT NOT NULL,
    PRIMARY KEY (role_id, permission_id),
    FOREIGN KEY (role_id) REFERENCES roles(id),
    FOREIGN KEY (permission_id) REFERENCES permissions(id)
);
```

### 4.5 交接记录表（permission_transfers）

```sql
CREATE TABLE permission_transfers (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    from_admin_id   BIGINT NOT NULL,   -- 被交接者（A）
    to_admin_id     BIGINT NOT NULL,   -- 接收者（B）
    initiated_by    BIGINT NOT NULL,   -- 交接发起人
    approved_by     BIGINT DEFAULT NULL, -- 审批人（NULL=待审批）
    permission_snapshot JSON NOT NULL,  -- JSON: 交接时的权限列表
    status          TINYINT NOT NULL DEFAULT 1, -- 1=待审批, 2=已批准, 3=已拒绝, 4=已取消
    approved_at     DATETIME DEFAULT NULL,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (from_admin_id) REFERENCES admins(id),
    FOREIGN KEY (to_admin_id) REFERENCES admins(id),
    FOREIGN KEY (initiated_by) REFERENCES admins(id),
    FOREIGN KEY (approved_by) REFERENCES admins(id)
);
```

### 4.6 操作日志表（admin_logs）

```sql
CREATE TABLE admin_logs (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    admin_id      BIGINT NOT NULL,
    action        VARCHAR(50) NOT NULL,   -- login/create/update/delete/transfer
    target_type   VARCHAR(30) DEFAULT NULL, -- admins/roles/permissions
    target_id     BIGINT DEFAULT NULL,
    detail        JSON,                     -- 变更前后详情
    ip            VARCHAR(45),
    user_agent    VARCHAR(500),
    created_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_admin (admin_id),
    INDEX idx_action (action),
    INDEX idx_created (created_at)
);
```

## 六、安全设计

### 6.1 密码安全

- bcrypt加密存储，salt rounds = 12
- 密码强度校验：12位以上，大小写+数字+特殊字符
- 连续5次登录失败，锁定30分钟
- 90天未修改密码，强制要求修改

### 6.2 会话安全

- Session有效期：8小时（ inactivity 30分钟自动过期）
- 支持同时登录设备数限制：管理员3台，超级管理员5台
- 新登录时自动失效旧Session（可选配置）

### 6.3 审计

- 所有操作记录IP、UA、时间
- 日志不可物理删除
- 超级管理员操作重点审计（登录邮件通知）
