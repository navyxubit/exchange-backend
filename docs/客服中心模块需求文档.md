## 一、模块概述

### 1.1 背景与目标

交易所后台需要一个完整的**智能客服中心**，融合：
- **MaxKB** 的 AI 知识库问答（RAG）能力
- **Chatwoot** 的在线人工客服与会话管理能力
- **工单系统**（自研）

实现：用户提问 → AI 优先回答（基于知识库）→ 无法解答时转人工 → 复杂问题生成工单 → 全链路可追溯。

### 1.2 技术选型依据

| 能力 | 来源 | 关键功能 |
|---|---|---|
| AI 知识库问答 | MaxKB | RAG 检索、文档上传、向量化、LangChain 工作流 |
| 在线客服会话 | Chatwoot | Inbox 多渠道、Conversation 状态机、Message 类型、Contact 管理 |
| 工单管理 | 自研 | 工单创建/分配/处理/关闭、优先级、抄送 |
| AI + 人工融合 | 自研整合层 | AI 无法回答时自动转人工、会话桥接、知识库学习 |

### 1.3 设计原则

- **AI 优先**：用户问题先由 AI 处理，降低人工客服压力
- **无缝转接**：用户无感知地从 AI 切换到人工，聊天记录完整保留
- **知识闭环**：AI 无法解答的问题，人工回答后可一键加入知识库
- **全链路可追溯**：每条消息、每次转接、每个工单都有记录
- **多渠道统一**：Web、App、API 等多入口统一管理

## 三、功能详情

---

### 3-A：AI 知识库（MaxKB 重构整合）

#### A1. 知识库管理

| 功能 | 说明 |
|---|---|
| 知识库创建 | 创建知识库，配置名称、描述、权限范围（共享/工作空间） |
| 文档上传 | 支持上传 PDF、Word、TXT、Markdown、Excel、CSV |
| 文档解析 | 自动拆分文档为段落（Chunk），支持手动标注段落 |
| 外部数据源 | 支持爬取指定 URL 网页内容（Web 站点类型） |
| 向量化 | 基于 Embedding 模型将段落转为向量，存储于 PostgreSQL + pgvector |
| 知识库分类 | 支持多级文件夹管理（树形结构） |
| 标签管理 | 给文档/段落打标签，便于精细化检索 |
| 文档审核 | 上传的文档需要审核后生效（可配置） |

#### A2. 检索与召回

| 功能 | 说明 |
|---|---|
| 混合检索 | 支持向量检索（语义相似）+ 关键词检索（BM25）的混合模式 |
| 重排序 | 对召回结果进行重排序（Rerank）提升相关性 |
| 命中处理 | 配置命中高相似度（>90%）时直接返回，不走 LLM 生成 |
| 多知识库检索 | 支持同时检索多个知识库，聚合结果 |
| 引用来源 | 返回答案时附带引用的文档段落，可点击查看原文 |

#### A3. 工作流编排（高级）

| 功能 | 说明 |
|---|---|
| 工作流画布 | 可视化拖拽编排 AI 对话流程 |
| LLM 调用节点 | 配置 LLM 模型（DeepSeek / Qwen / GPT 等） |
| 知识库检索节点 | 在工作流中插入 RAG 检索节点 |
| 条件分支 | 根据用户意图/变量值走不同分支 |
| 函数调用 | MCP 工具调用能力，扩展 AI 行动边界 |
| 开场白配置 | AI 客服开场白、推荐问题 |

#### A4. 模型配置

| 功能 | 说明 |
|---|---|
| LLM 配置 | 对接各种大模型（本地私有 DeepSeek / 公共 API 等） |
| Embedding 模型 | 配置向量化模型 |
| TTS / STT | 可选配置语音合成/识别（语音客服场景） |
| 模型参数 | Temperature / Top-P / Max Tokens 等参数配置 |

### 3-B：AI 客服机器人（融合层）

#### B1. 智能问答

| 功能 | 说明 |
|---|---|
| FAQ 问答 | 基于知识库 RAG 回答用户常见问题 |
| 多轮对话 | 支持上下文多轮对话（Session 管理） |
| 意图识别 | 识别用户意图（咨询/投诉/建议/转账等） |
| 情绪识别 | 识别用户情绪（正面/中性/负面），负面时优先转人工 |
| 寒暄处理 | 支持大模型自带寒暄能力 |
| 拒答 | 涉及敏感内容（政治/色情/暴力等）时统一拒答 |

#### B2. 智能转人工

| 功能 | 说明 |
|---|---|
| 自动转人工规则 | 配置触发条件：用户输入"转人工"、情绪负面、连续3次未命中知识库等 |
| 转接提示 | 转人工前 AI 告知用户"正在为您转接人工客服，请稍候" |
| 会话桥接 | AI 对话记录完整传递给人工坐席，无需用户重复描述问题 |
| 手动转人工 | 用户随时输入"人工" / "客服" 触发转接 |
| 转接分配 | 支持按轮询/负载均衡分配给在线坐席 |

#### B3. 知识库学习

| 功能 | 说明 |
|---|---|
| 未知问题标记 | AI 无法解答的问题自动标记为"未知问题" |
| 人工回复学习 | 人工客服回答后，可一键"加入知识库"，自动创建 FAQ |
| 问题优化 | 将用户表述的口语化问题优化为标准 FAQ 问法 |
| 冷启动 | 支持批量导入 FAQ 初始数据 |

### 3-C：在线人工客服（Chatwoot 重构整合）

#### C1. 渠道管理（Inbox）

| 渠道 | 说明 | 整合方式 |
|---|---|---|
| Web Widget | 网页嵌入式客服窗口 | 自研前端嵌入 MaxKB AI + Chatwoot Widget |
| 移动端 App | App 内客服入口 | 移动端 SDK 接入 |
| API | 第三方系统接入 | REST API 接收用户消息 |
| 工单 | 用户提交的表单问题 | 自研工单系统 → Chatwoot Conversation |

> **注**：Email / WhatsApp / Telegram / Facebook 等国外渠道暂不接入。如后续需扩展，可按 Chatwoot Channel 架构添加。

#### C2. 会话管理（Conversation）

| 功能 | 说明 |
|---|---|
| 会话状态 | open（进行中）/ resolved（已解决）/ pending（等待中）/ snoozed（稍后提醒） |
| 会话优先级 | low / medium / high / urgent |
| 会话分配 | 手动分配 / 自动分配（轮询、负载均衡、最少承接） |
| 团队分配 | 按技能组（充值问题→财务组、合约问题→风控组）分配 |
| 会话合并 | 同一用户多窗口会话可合并 |
| 会话置顶/标记 | 重要会话可置顶或加标记色 |
| 回复模板 | 预设快捷回复模板（Macros） |
| 敏感信息脱敏 | 手机号/邮箱/银行卡号在聊天中脱敏展示 |

#### C3. 消息管理（Message）

| 消息类型 | 说明 |
|---|---|
| incoming | 用户发送的消息 |
| outgoing | 客服/AI 发送的消息 |
| activity | 系统消息（状态变更、转接等） |
| template | 模板消息（如发送卡片、表单） |

消息支持：文字、图片、文件、音频、视频、卡片、表单。

#### C4. 联系人管理（Contact）

| 功能 | 说明 |
|---|---|
| 访客 /  Lead / 客户 | Contact 类型区分 |
| 用户画像 | 关联交易所用户ID、注册时间、VIP等级、持仓量 |
| 历史会话 | 查看该联系人的所有历史会话记录 |
| 备注标签 | 客服可给联系人打标签（高价值/投诉倾向等） |
| 黑名单 | 恶意骚扰用户可加入黑名单 |

#### C5. 客服坐席管理

| 功能 | 说明 |
|---|---|
| 坐席列表 | 客服账号管理（与系统管理员模块共用 Admin 表） |
| 在线状态 | 在线 / 离开 / 忙碌 / 离线 |
| 接待上限 | 单个坐席同时最大接待会话数（可配置，默认 10） |
| 接待统计 | 各坐席今日/本周承接会话数、响应时长、解决率 |

### 3-D：工单管理（自研）

#### D1. 工单创建

| 触发方式 | 说明 |
|---|---|
| 用户提交 | 用户从 App/Web 客服入口提交工单（联系BD、活动申请、问题反馈等） |
| AI 转接 | AI 无法解答时生成工单，转给对应客服组 |
| 客服升级 | 人工客服处理中的会话可升级为正式工单 |
| 内部创建 | 管理员/客服在后台手动创建工单 |

#### D2. 工单字段

| 字段 | 说明 |
|---|---|
| 工单编号 | 唯一编号（格式：TK-年月日-序号） |
| 标题 | 问题简述 |
| 类型 | 咨询 / 投诉 / 建议 / Bug / 其他 |
| 优先级 | 低 / 中 / 高 / 紧急 |
| 状态 | 待处理 / 处理中 / 待用户确认 / 已解决 / 已关闭 |
| 提交人 | 用户信息（关联交易所用户） |
| 受理人 | 当前处理该工单的客服 |
| 关联会话 | 关联的 Chatwoot 会话 ID（如有） |
| 关联知识库 | 可关联相关知识库段落 |
| 附件 | 用户上传的截图、文件 |
| 内部备注 | 仅客服/管理员可见的备注 |
| 处理记录 | 工单变更时间线（创建→分配→处理→关闭） |

#### D3. 工单流程

```
用户提交工单
    → 客服组长/系统自动分配（按类型/优先级/负载）
    → 受理人接单（状态→处理中）
    → 处理中（可添加内部备注、关联知识库、关联会话）
    → 需要用户确认 → 等待用户确认
    → 已解决 → 用户确认 → 已关闭
    → 超时未响应 → 自动升级/提醒
```

#### D4. 工单处理工具

| 功能 | 说明 |
|---|---|
| 回复模板 | 预设回复模板，快速填充 |
| 内部协作 | @ 同事协助，处理人可临时转交 |
| 工单抄送 | 工单处理结果可抄送指定人员（邮件/站内） |
| 满意度评价 | 工单关闭后邀请用户评价（1-5星） |
| 批量处理 | 批量修改工单状态/受理人/优先级 |

## 四、数据模型

### 4.1 知识库相关（来自 MaxKB，重构）

```sql
-- 知识库
CREATE TABLE kb_knowledge (
    id              CHAR(64) PRIMARY KEY,
    name            VARCHAR(150) NOT NULL,
    workspace_id    CHAR(64) NOT NULL DEFAULT 'default',
    type            INT NOT NULL DEFAULT 0,  -- 0=通用, 1=Web, 2=飞书等
    scope           VARCHAR(20) DEFAULT 'WORKSPACE',
    desc            VARCHAR(256),
    embedding_model_id  CHAR(64),
    user_id         CHAR(64),
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 文档
CREATE TABLE kb_document (
    id              CHAR(64) PRIMARY KEY,
    knowledge_id    CHAR(64) NOT NULL,
    name            VARCHAR(150) NOT NULL,
    char_length     INT DEFAULT 0,
    status          VARCHAR(20) DEFAULT '000',
    is_active       BOOLEAN DEFAULT TRUE,
    type            INT DEFAULT 0,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 段落
CREATE TABLE kb_paragraph (
    id              CHAR(64) PRIMARY KEY,
    document_id     CHAR(64) NOT NULL,
    knowledge_id    CHAR(64) NOT NULL,
    content         VARCHAR(102400) NOT NULL,
    title           VARCHAR(256) DEFAULT '',
    status          VARCHAR(20) DEFAULT '000',
    hit_num         INT DEFAULT 0,
    is_active       BOOLEAN DEFAULT TRUE,
    position        INT DEFAULT 0,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 向量（pgvector）
CREATE TABLE kb_embedding (
    id              CHAR(64) PRIMARY KEY,
    source_id      CHAR(64) NOT NULL,
    source_type     INT DEFAULT 0,  -- 0=Problem, 1=Paragraph, 2=Title
    knowledge_id    CHAR(64) NOT NULL,
    document_id     CHAR(64),
    paragraph_id    CHAR(64),
    embedding       VECTOR(1536),  -- 根据模型维度
    is_active       BOOLEAN DEFAULT TRUE
);

-- 标签
CREATE TABLE kb_tag (
    id              CHAR(64) PRIMARY KEY,
    knowledge_id    CHAR(64) NOT NULL,
    tag_key         VARCHAR(64) NOT NULL,
    tag_value       VARCHAR(128) NOT NULL,
    UNIQUE(knowledge_id, tag_key, tag_value)
);

-- 文档标签关联
CREATE TABLE kb_document_tag (
    document_id     CHAR(64) NOT NULL,
    tag_id          CHAR(64) NOT NULL,
    PRIMARY KEY(document_id, tag_id)
);
```

### 4.2 AI 对话（来自 MaxKB，重构）

```sql
-- AI 对话会话（来自 MaxKB ApplicationChat，重构整合）
CREATE TABLE ai_chat_session (
    id              CHAR(64) PRIMARY KEY,
    application_id  CHAR(64) NOT NULL,      -- AI应用ID
    user_id         BIGINT,                  -- 交易所用户ID（可为NULL匿名）
    user_type       VARCHAR(20) DEFAULT 'ANONYMOUS_USER',  -- ANONYMOUS_USER/CHAT_USER/SYSTEM_API_KEY
    session_type    VARCHAR(20) DEFAULT 'AI',  -- AI/人工/工单
    abstract        VARCHAR(1024) DEFAULT '',
    status          VARCHAR(20) DEFAULT 'open',  -- open/resolved/pending
    source          JSON DEFAULT '{}',        -- 来源：Web/App/API
    ip_address      VARCHAR(128),
    star_num        INT DEFAULT 0,
    trample_num     INT DEFAULT 0,
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- AI 对话记录（来自 MaxKB ApplicationChatRecord，重构整合）
CREATE TABLE ai_chat_message (
    id              CHAR(64) PRIMARY KEY,
    chat_session_id CHAR(64) NOT NULL,
    sender_type     VARCHAR(20) NOT NULL,   -- user/ai/human
    sender_id       BIGINT,                 -- 发送者ID（用户/坐席ID）
    message_type    VARCHAR(20) NOT NULL,   -- text/image/file/activity/template
    content         TEXT,                    -- 消息内容
    ai_source       VARCHAR(50),            -- AI来源：knowledge_base/llm_direct/human
    reference_docs  JSON,                   -- AI回答时引用的知识库段落
    vote_status     VARCHAR(10) DEFAULT '-1', -- un_vote/star/trample
    vote_reason     VARCHAR(50),
    index           INT NOT NULL,           -- 消息序号
    details         JSON,                   -- 扩展详情（Token消耗、运行时长等）
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- 未知问题记录（AI无法回答的问题）
CREATE TABLE ai_unknown_question (
    id              CHAR(64) PRIMARY KEY,
    chat_session_id CHAR(64),
    question        VARCHAR(10240) NOT NULL,
    hit_count       INT DEFAULT 1,
    status          VARCHAR(20) DEFAULT 'pending',  -- pending/resolved/ignored
    suggested_answer VARCHAR(10240),  -- 人工补充的建议答案
    resolved_by     CHAR(64),         -- 解决该问题的人工客服ID
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

### 4.3 工单（自研）

```sql
-- 工单表
CREATE TABLE ticket (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    ticket_no       VARCHAR(32) NOT NULL UNIQUE,  -- TK-20260415-0001
    title           VARCHAR(256) NOT NULL,
    type            VARCHAR(20) NOT NULL,   -- consultation/complaint/suggestion/bug/other
    priority        VARCHAR(10) NOT NULL DEFAULT 'medium',  -- low/medium/high/urgent
```

## 五、AI + 人工融合流程

### 5.1 完整用户会话流程

```
用户进入客服
    ↓
AI 接待（开场白 + 知识库检索）
    ↓
┌─────────────────────┐
│ 用户提问             │
│   ↓                  │
│ 知识库检索命中？      │
│   ↓       ↓          │
│  是       否          │
│   ↓       ↓          │
│ AI回答    AI尝试优化重答 │
│   ↓       ↓          │
│  命中？    命中？     │
│   ↓  是   ↓ 否       │
│  直接返回  转人工     │
└─────────────────────┘
    ↓
┌─ 用户满意 → 会话关闭
├─ 用户仍有问题 → 继续 AI 问答
├─ 用户输入"转人工" / 情绪负面 / 连续未命中 → 转人工
└─ 问题复杂 → 生成工单，人工会话处理
```

### 5.2 转接细节

| 触发条件 | 动作 |
|---|---|
| 用户输入"人工"/"客服" | 直接转接在线坐席 |
| 情绪负面检测 | AI 先安抚，同步转接人工 |
| 知识库连续3次未命中 | 自动转接人工 |
| 涉及资产/转账操作 | 直接转人工（安全优先） |
| 用户提交表单/附件 | 生成工单，分配给对应客服组 |

### 5.3 会话桥接

- AI 对话记录（Chatwoot Message type=activity）标注"AI转人工"
- 人工坐席看到的界面：保留完整 AI 对话 + 用户最后一条消息高亮
- 坐席可一键将本次人工回答加入知识库

### 3-H：多语言实时翻译

#### H1. 设计原则

- **客服独立配置**：每个客服可独立设置自己的偏好语言
- **用户无感知**：用户始终使用自己的语言，翻译在后台对客服可见
- **双向翻译**：用户→客服 显示原文+译文；客服→用户 显示译文+（可选）原文

#### H2. 语言配置

| 角色 | 配置项 |
|---|---|
| 客服 | 可选择：中文 / 英文 / 日文 / 韩文 等（多选，主语言排首位） |
| 用户 | 自动识别（根据用户设置或发送内容检测语言） |
| 系统翻译 | 接入在线大模型（DeepSeek / GPT / 谷歌翻译）或本地模型 |

#### H3. 翻译显示规则

**场景1：用户（中国用户）→ 客服（英文配置）**
```
用户发送（中文）："我的充值还没到账"

客服端显示：
  "我的充值还没到账"        ← 原文
  "My recharge hasn't arrived yet" ← 译文（AI实时翻译）
```

**场景2：客服（英文配置）→ 用户（中国用户）**
```
客服发送（英文）："I've checked your transaction, it's processing."

用户端显示：
  "我已检查您的交易，正在处理中。" ← 译文
  (可选) "I've checked..."        ← 原文（可折叠）
```

#### H4. 翻译技术方案

| 方案 | 优点 | 缺点 |
|---|---|---|
| 在线大模型（DeepSeek/GPT） | 翻译质量高，支持多语言 | 有成本，需网络 |
| 本地部署模型 | 隐私好，无成本 | 占用服务器资源，初始化慢 |
| Google Translate API | 支持语言多，速度快 | 有成本，需API Key |

**推荐**：优先使用在线大模型（DeepSeek R1 等），可配置切换。

#### H5. 实现细节

- 翻译异步进行，不阻塞消息发送
- 翻译缓存：相同内容短期不重复翻译
- 翻译状态标注：显示"原文" / "AI翻译" 标识
- 翻译异常时：降级显示原文，不阻塞会话

### 3-I：Flutter 端接入方案

#### I1. 推荐方案：WebView 嵌入式

Flutter App 推荐使用 **WebView 嵌入式**方案，理由如下：

| 考量 | 说明 |
|---|---|
| 开发成本低 | 复用 Web 端聊天界面，无需单独开发 Flutter 原生 UI |
| 迭代同步 | Web 端更新后 App 立即生效，无需发版 |
| 多端一致 | iOS / Android / Web 三端体验完全一致 |
| WebSocket 支持 | 主流 WebView（flutter_inappwebview）完整支持 WebSocket |

#### I2. 推荐技术栈

| 库 | 用途 |
|---|---|
| `flutter_inappwebview` | 增强型 WebView，支持 WebSocket、JS 交互 |  | `flutter_local_notifications` | 新消息桌面通知 |
| `uni_links` / `app_links` | 深度链接（从通知打开特定会话） |

#### I3. WebView 集成架构

```
Flutter App
└── InAppWebView（占满全屏或半屏客服界面）
    ├── 加载客服中心 H5 页面
    ├── WebSocket 连接（实时消息）
    ├── JS Bridge（Flutter ↔ H5 通信）
    │   ├── 登录态注入（用户Token）
    │   ├── 推送通知触发（Flutter本地通知）
    │   ├── 页面切换控制（打开/关闭客服）
    │   └── 语言配置读取
    └── 翻译功能（H5 页面内置）
```

#### I4. 关键集成点

| 功能 | 实现方式 |
|---|---|
| 用户身份认证 | 启动 WebView 时通过 JS 注入 JWT Token |
| 新消息通知 | H5 通过 `postMessage` 通知 Flutter，触发本地通知 |
| 退出客服 | H5 调用 Flutter 方法关闭 WebView |
| 打开指定会话 | Deep Link 格式：`exchange://chat?session_id=xxx` |
| 拍照/上传图片 | H5 调用 Flutter 相机/相册插件 |
| 主题适配 | H5 读取系统深色/浅色模式 |

#### I5. 替代方案：纯 Dart SDK

如未来需要更原生体验，可封装纯 Dart SDK：

```
// 方案：封装 REST API + WebSocket Dart 客户端
packages:
  web_socket_channel: ^2.0.0   # WebSocket 连接
  dio: ^5.0.0                  # REST API 请求
  flutter_secure_storage:        # 敏感信息存储

// SDK 结构
lib/
  src/
    customer_service_sdk.dart   # 主入口
    api/
      auth_api.dart            # 认证
      conversation_api.dart    # 会话
      message_api.dart         # 消息
    websocket/
      socket_manager.dart       # WebSocket管理
      message_handler.dart      # 消息处理
    translation/
      translator.dart           # 翻译服务
    models/
      session.dart
      message.dart
      ticket.dart
```

> **建议**：初期采用 WebView 方案快速上线，后期按需封装纯 Dart SDK。
