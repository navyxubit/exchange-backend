## 一、模块概述

### 1.1 定位

财务中心是交易所后台管理系统中负责**资金管理、账务核算、财务报表**的核心模块。面向财务人员、管理层提供全面的资金可视化和操作能力。

### 1.2 设计原则

- **资金安全优先**：所有资金操作需多重校验，支持审批流
- **账目不可篡改**：财务流水仅追加（Append-Only），不支持修改和删除
- **职责分离**：操作员、审核员、管理员三级权限相互制约
- **可追溯**：所有操作留痕，可完整回溯每一笔资金流向

#### C2. 提现审核（核心安全流程）

**审核层级：**

| 提现金额区间 | 审核要求 |
|---|---|
| 小额（< $1,000） | 单审核员通过即可 |
| 中额（$1,000 - $10,000） | 双审核员复核 |
| 大额（> $10,000） | 三审核员会签 + 财务主管审批 |
| 超级大额（> $100,000） | 额外需CEO/COO审批 |

**审核操作：**
- 审核通过 → 进入打币队列
- 审核拒绝 → 附原因，资产释放回用户账户
- 标记可疑 → 进入反洗钱调查流程

**安全校验：**
- 邮箱/手机/二次验证状态
- 是否为白名单地址
- 是否为异常时间段操作
- 是否触发反洗钱规则

#### C3. 提现白名单管理

| 功能 | 说明 |
|---|---|
| 白名单列表 | 用户添加的信任提现地址 |
| 白名单审核 | 新增地址需审核（可配置直接通过） |
| 免审额度 | 白名单地址单笔免审上限（可配置） |
| 单日提现上限 | 用户自设 + 平台上限双重限制 |

#### C4. 打币操作（热钱包出币）

| 功能 | 说明 |
|---|---|
| 打币队列 | 审核通过的提现进入打币队列 |
| 批量打币 | 支持合并多笔一起打币（节省gas） |
| 签名机制 | 冷签名（离线签名）/ 热签名（自动） |
| 交易加速 | 支持提高gas加速pending交易 |
| 打币限额 | 单笔/单日打币上限（超过需多签） |

#### C5. 提现统计分析

| 指标 | 说明 |
|---|---|
| 今日提现笔数 / 总额 | 按币种分类 |
| 审核队列 | 待审核笔数、金额 |

### 3-E：钱包管理

#### E1. 钱包架构

```
平台钱包体系
├── 热钱包（Hot Wallet）
│   ├── 日常运营用：用户充值收款、用户提现打币
│   ├── 自动归集：从地址池定期归集
│   └── 限额控制：单钱包余额上限
├── 冷钱包（Cold Storage）
│   ├── 长期存储：95%+资产离线存储
│   ├── 离线签名：物理隔离，无网络连接
│   └── 多签机制：M/N多签审批
├── 运营账户（Operation Account）
│   ├── 平台自有资金
│   ├── 手续费收入
│   └── 风险准备金
└── 归集账户（Collection Account）
    └── 地址池资金归集中转
```

#### E2. 热钱包管理

| 功能 | 说明 |
|---|---|
| 余额监控 | 实时余额、各币种分布 |
| 自动归集 | 定时将地址池余额归集到热钱包 |
| 余额预警 | 超过/低于阈值时告警 |
| 充值监控 | 大额充值实时告警 |
| 出币限额 | 单笔/日累计出币上限 |
| 补币机制 | 热钱包余额不足时从冷钱包补充 |

#### E3. 冷钱包管理

| 功能 | 说明 |
|---|---|
| 冷钱包地址 | 地址公开（可审计），私钥完全离线 |
| 多签审批 | 大额转出需M/N多签（配置：如3/5多签） |
| 补币审批 | 热钱包从冷钱包补充需多签审批 |
| 冷钱包查看 | 纯查看权限，财务/审计人员可查看余额 |
| 地理位置 | 冷钱包存储地点（物理位置保密，仅超级管理员可知） |

#### E4. 钱包操作日志

| 操作类型 | 记录内容 |

### 3-H：资产核对

#### H1. 链上 vs 链下核对

| 核对项 | 说明 |
|---|---|
| 平台链上余额 | 从区块链读取的热钱包 + 冷钱包余额 |
| 平台数据库余额 | 数据库中各币种钱包账户余额总和 |
| 差异值 | 链上余额 - 数据库余额 |
| 差异告警 | 差异值不为0时触发告警 |

#### H2. 用户账户核对

| 核对项 | 说明 |
|---|---|
| 用户资产总和 | 数据库所有用户各币种余额之和 |
| vs 平台托管资产 | 用户资产总和 vs 平台实际托管资产 |
| 差异处理 | 差异需在当天结账前查明原因并处理 |

#### H3. 对账任务

| 功能 | 说明 |
|---|---|
| 自动对账 | 每小时执行一次自动对账任务 |
| 手动对账 | 支持财务人员手动触发对账 |
| 对账报告 | 生成对账结果报告，含差异明细 |
| 差异处理 | 差异进入人工排查流程，记录处理结果 |

#### H4. 平账流程

```
发现差异 → 差异分析（链上重放？数据库错误？人为攻击？）→ 定位原因
    → 小额：直接平台承担，手动调账平账
    → 大额：冻结相关账户，排查原因后处理
    → 可疑：进入安全/法务调查流程
    → 记录：对账日志、处理记录存档
```

## 五、数据模型（核心表）

### 5.1 钱包表（wallets）

```sql
CREATE TABLE wallets (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    coin          VARCHAR(20)  NOT NULL,          -- 币种代码
    type          VARCHAR(20)  NOT NULL,          -- hot/cold/operation/collection
    balance       DECIMAL(30,8) NOT NULL DEFAULT 0, -- 余额
    frozen_balance DECIMAL(30,8) NOT NULL DEFAULT 0, -- 冻结余额
    address       VARCHAR(255) DEFAULT NULL,       -- 钱包地址（热钱包有）
    threshold     DECIMAL(30,8) DEFAULT NULL,     -- 余额预警阈值
    created_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_coin_type (coin, type)
);
```

### 5.2 充值记录表（deposits）

```sql
CREATE TABLE deposits (
    id              BIGINT PRIMARY KEY AUTO_INCREMENT,
    order_no        VARCHAR(64) NOT NULL UNIQUE,  -- 订单号
    user_id         BIGINT NOT NULL,
    coin            VARCHAR(20) NOT NULL,
    amount          DECIMAL(30,8) NOT NULL,       -- 充值数量
    net_amount       DECIMAL(30,8) NOT NULL,       -- 到账数量（扣网络费）
    fee             DECIMAL(30,8) NOT NULL DEFAULT 0, -- 网络手续费
    address         VARCHAR(255) NOT NULL,         -- 充值地址
    txid            VARCHAR(255) DEFAULT NULL,
    block_height    BIGINT DEFAULT NULL,
    confirmations   INT DEFAULT 0,
    status          TINYINT NOT NULL DEFAULT 0,    -- 0=待确认,1=已到账,2=已取消,3=失败
    from_address    VARCHAR(255) DEFAULT NULL,     -- 来源地址
    created_at      DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    confirmed_at    DATETIME DEFAULT NULL,
    INDEX idx_user (user_id),
    INDEX idx_coin_status (coin, status),
    INDEX idx_created (created_at)
);
```
